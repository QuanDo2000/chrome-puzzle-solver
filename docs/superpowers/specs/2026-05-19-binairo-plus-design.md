# Binairo Plus support — design spec

**Date:** 2026-05-19
**Status:** Approved, pending implementation plan
**Scope:** Add support for the Binairo Plus variant on
`puzzles-mobile.com/binairo-plus/*`, served by the same handler/solver
infrastructure as standard Binairo with a new rule for the comparison-clue
markers (`=` / `≠` between adjacent cells).

## Goals

1. Detect Binairo Plus puzzles on `/binairo-plus/*` and run the full widget
   surface (Solve, Hint, Loop, Undo, Redo, Dump, canvas preview).
2. Extend `BinairoSolver` with a `_applyComparison` propagation rule so
   comparison clues are first-class deduction inputs alongside no-triples,
   balance, uniqueness, and single-remaining.
3. Render the `=` / `≠` markers on the preview canvas so the user can
   verify hints against the constraint set.
4. Update the "Supported puzzles" list shown on the homepage.

## Non-goals

- A separate `BinairoPlusSolver` class. The user chose to extend the existing
  solver with an optional `comparisonClues` parameter; empty clues = standard
  Binairo behaviour with zero-cost (the new rule no-ops with no constraints).
- Backwards-incompatible changes to the existing solver or handler API. All
  new parameters are optional with safe defaults.

## Recon facts (from a live 6x6-easy capture, 2026-05-19)

- **URL:** `https://www.puzzles-mobile.com/binairo-plus/random/6x6-easy`.
- **`Game.slug === 'binairo-plus'`**; otherwise the engine reuses Binairo's
  field names (`task`, `currentState.cellStatus`, `puzzleWidth/Height`,
  `saveState`, `redraw`).
- **`Game.task`** is the 2D givens array in the same `-1/0/1` encoding
  (`-1=blank, 0=given-zero, 1=given-one`).
- **`Game.currentState.cellStatus`** is the same `0/1/2` encoding
  (`0=empty, 1=one, 2=zero`).
- **`Game.comparisonClues`** is a sparse 2D array of flag integers. Decoded
  via the engine's `decode_flag` against four exported constants:
  `FLAG_RIGHT_EQ=1`, `FLAG_RIGHT_NE=2`, `FLAG_DOWN_EQ=4`, `FLAG_DOWN_NE=8`.
  Flags are OR-able. Sample value `10 = 8 | 2` decodes to "down ≠ AND
  right ≠" on that cell. A flag at `(r, c)` describes the relation between
  `(r, c)` and `(r, c+1)` (RIGHT_* bits) and/or between `(r, c)` and
  `(r+1, c)` (DOWN_* bits).
- The page renders the markers as small `=` and `≠` glyphs between cells.

### `BinairoSolver` changes

**Constructor signature** becomes:

```js
new BinairoSolver({
  rows, cols, givens,
  initialState?,        // existing
  comparisonClues?,     // NEW: sparse 2D of flag integers (page-native shape)
})
```

`comparisonClues` defaults to `[]`. The constructor immediately normalizes
it into a flat canonical form for the rule loop:

```js
this.compConstraints = this._decodeComparison(comparisonClues);
//   [{ aR, aC, bR, bC, sameSign: boolean }, ...]
//   sameSign === true  → cells A and B must be equal
//   sameSign === false → cells A and B must differ
```

`_decodeComparison` walks the sparse 2D and emits one entry per set flag
bit. A cell with flag `10` (= `8|2`) emits two entries: one right-NE and
one down-NE. Boundary clues that point off-grid (e.g. RIGHT_* on the last
column) are silently dropped.

**Propagation rule.** Added to `propagate()` between `_applyBalance` and
`_applyUniqueness` — the comparison rule is cheap and unblocks lines that
the existing rules can't:

```js
_applyComparison(onChange) {
  for (const k of this.compConstraints) {
    const a = this._get(k.aR, k.aC);
    const b = this._get(k.bR, k.bC);
    if (a !== 0 && b !== 0) {
      const equal = a === b;
      if (equal !== k.sameSign) return false; // contradiction
      continue;
    }
    if (a === 0 && b === 0) continue; // can't force yet
    const known = a !== 0 ? a : b;
    const target = k.sameSign ? known : (known === 1 ? 2 : 1);
    const r = a !== 0 ? k.bR : k.aR;
    const c = a !== 0 ? k.bC : k.aC;
    if (this._wouldCreateTriple(r, c, target)) return false;
    if (this._assign(r, c, target)) onChange();
  }
  return true;
}
```

Because `_applyComparison` validates every both-sides-known pair every
pass, a successful `propagate()` guarantees no comparison violations
remain. No separate completion-time check is needed (the existing
`_hasDuplicateLines` is still called for uniqueness, which has a
genuine gap; comparison has no such gap).

**Cache key.** `_cacheKey` mixes the comparison clues into FNV-1a after
the givens — a binairo and a binairo-plus board with identical givens
must produce distinct cache keys. The sparse shape is hashed cell-by-cell
in row-major order; missing entries hash as `0`.

**Test helper.** `BinairoSolver.compConstraintsFromFlags(comparisonClues)`
exposes the decoder as a static so tests can construct constraints
directly instead of building sparse 2D arrays.

### Handler unification

`binairoHandler.matches()` becomes:

```js
matches() {
  return isPuzzlesMobilePage() && (
    window.location.pathname.includes('/binairo/') ||
    window.location.pathname.includes('/binairo-plus/')
  );
},
```

`detect()` drops the "Binairo comparison-clue variant not yet supported"
refusal entirely. The returned puzzle context gains a `comparisonClues`
field carrying whatever `readBinairoData` produced (empty array for
standard binairo, populated sparse 2D for plus).

`puzzleData.type` stays `'binairo'` regardless of the path — the rule set
is a superset, the path is a UI distinction only.

### `content.js` changes

**`solveExtraData`** for binairo:

```js
if (detectedGrid?.type === 'binairo') {
  return {
    rows: detectedGrid.rows,
    cols: detectedGrid.cols,
    givens: detectedGrid.givens,
    comparisonClues: detectedGrid.comparisonClues || [],
  };
}
```

**`binairoCacheKey`** folds `data.comparisonClues` into the FNV hash after
the givens loop, byte-by-byte in row-major order with a leading `null`
sentinel mix so an empty/missing clue grid still produces a stable key.

**`drawPreview` `staticLayer`** renders the `=` / `≠` glyphs:

- For each non-null `(r, c, flag)` in `puzzleData.comparisonClues`,
  decode the four bits and draw at the cell-boundary midpoints:
  - `RIGHT_EQ` (1) at `x = (c+1)*cellSize`, `y = r*cellSize + cellSize/2` — glyph `=`.
  - `RIGHT_NE` (2) same position — glyph `≠`.
  - `DOWN_EQ` (4) at `x = c*cellSize + cellSize/2`, `y = (r+1)*cellSize` — glyph `=`.
  - `DOWN_NE` (8) same position — glyph `≠`.
- Glyph style: dark grey (`#1f2937`), font size `cellSize * 0.45`. A
  small white outline (`strokeStyle = '#fff'`, `lineWidth = 2`) is drawn
  first so the glyph stays legible over disc fills painted in the
  dynamic layer below.
- Lives in `staticLayer` — rebuilt only when puzzle shape changes,
  zero per-tick cost.

**Worker dispatch.** `solver.worker.js`'s binairo arm passes
`comparisonClues` through:

```js
} else if (type === 'binairo' && extraData) {
  const s = new BinairoSolver({
    rows: extraData.rows,
    cols: extraData.cols,
    givens: extraData.givens,
    comparisonClues: extraData.comparisonClues || [],
    initialState: initialGrid || null,
  });
  result = s.solve();
}
```

**Supported puzzles list:**

```js
const SUPPORTED_PUZZLES = [
  // ...
  { name: 'Binairo',      url: 'https://www.puzzles-mobile.com/binairo/' },
  { name: 'Binairo Plus', url: 'https://www.puzzles-mobile.com/binairo-plus/' },
];
```

### Tests, fixtures, benches

`tests/fixtures/puzzles.js` gets a deterministic `binairoPlus6x6` entry —
shape `{ rows, cols, givens, comparisonClues }`. Captured live from
`/binairo-plus/random/6x6-easy` during implementation.

`tests/capture.js`'s `solveBinairo` passes `comparisonClues` from the
fixture into the constructor. `tests/golden.js` regenerates and gains
a `binairoPlus6x6` snapshot.

`tests/solver.test.js` adds:
- A test that constructs the simplest possible board with a single
  RIGHT_EQ between two cells, places one, asserts the other is forced.
- The same shape with RIGHT_NE, asserts opposite forced.
- An inconsistent prefill that should cause `propagate()` to return
  false (contradiction).
- A golden-match test for `binairoPlus6x6`.

`tests/binairo-fuzz.test.js` extends `verifyBinairoRules` to accept an
optional `comparisonClues` and verify the constraints on the returned
grid. Adds a *constructive* fuzz block at 4×4 and 6×6: start from a
solved grid, sample a random subset of borders, attach a random `=`/`≠`
flag consistent with the solved values; build givens by knocking out
some cells; assert the solver returns a grid satisfying both the
standard rules and the comparison constraints.

`tests/fixtures/real-puzzles.js` gains `binairoPlusReal6x6_a` so
`bench-binairo.js` reports a median time for the plus path too. The
bench script's existing loop filters by `type === 'binairo'` and now
spreads `comparisonClues` into the constructor call.

`CLAUDE.md` gets a "Binairo Plus / comparison-clue support" subsection:
- Handler matches both `/binairo/` and `/binairo-plus/`.
- Flag encoding (`1=R-EQ, 2=R-NE, 4=D-EQ, 8=D-NE`, OR-able).
- `_decodeComparison` flattens to `compConstraints`; `_applyComparison`
  fires between balance and uniqueness.
- `comparisonClues` is the FOURTH field that distinguishes Binairo Plus
  from Binairo — the `puzzleData.type === 'binairo'` for both,
  differentiation lives in the data, not the type tag.

## Data flow (solve + hint, end-to-end)

1. User navigates to `/binairo-plus/random/...`. Content script loads.
   `binairoHandler.matches()` returns true.
2. User clicks Solve. `runSolve` → `handler.detect` → MAIN-world
   `readBinairoData` returns `{ task, width, height, comparisonClues }`.
3. `solveExtraData` packages `{ rows, cols, givens, comparisonClues }`
   and `runSolve` posts to the Blob-built worker.
4. Worker constructs `BinairoSolver` with the clues, runs
   `solve()` (local rules including `_applyComparison`, then lookahead
   if needed, then backtrack), returns the solved cellStatus grid.
5. `applySolveResult` caches, draws preview (with `=`/`≠` glyphs in
   the static layer), shows confirm-mode UI.
6. Apply → `handler.applySolution` → MAIN-world `applyBinairoState`
   → save+render ladder. Unchanged.

Hint flow follows the same getHint pipeline (local rules including
`_applyComparison` then line-restricted lookahead fallback).

## Error paths

- `readBinairoData` timeout → handler returns `error: 'No Binairo task data found'`.
- Invalid clues that contradict the givens → solver returns
  `{ solved: false, error: 'contradiction on initial propagation' }`.
- Invalid clues that point off-grid → silently dropped during decode.
- Comparison-rule contradiction discovered mid-backtrack → branch fails,
  backtracking continues.
- Apply path is unchanged — `applyBinairoState` only writes
  `cellStatus`; comparison clues are read-only on the page.

## Implementation phasing (suggested for the plan)

1. **Solver:** `_decodeComparison`, `_applyComparison`, `_cacheKey` mix-in.
   Unit tests for each. No external integration yet.
2. **Worker + handler:** thread `comparisonClues` through worker dispatch
   and handler.detect.
3. **`content.js`:** `solveExtraData`, `binairoCacheKey`, supported-puzzles
   entry.
4. **Preview rendering:** `=`/`≠` glyphs in staticLayer.
5. **Fixture + golden + bench:** capture a real puzzle via Dump, add to
   fixtures, regenerate goldens, verify bench runs.
6. **Fuzz coverage:** constructive comparison-clue fuzz.
7. **CLAUDE.md:** document the new subsection.

## Open assumptions to verify at impl time

1. **Flag bit values.** Recon shows the integer values `2`, `4`, `10` and
   the constants `FLAG_RIGHT_EQ, FLAG_RIGHT_NE, FLAG_DOWN_EQ, FLAG_DOWN_NE`.
   The mapping `1, 2, 4, 8` is the natural bit assignment consistent with
   `10 = 8 | 2`, but we should confirm during fixture capture (e.g. by
   probing `window.Game.FLAG_RIGHT_EQ` etc. in DevTools).
2. **Sparse-row interior.** `comparisonClues[r]` is sometimes `[null,
   null, x]` and sometimes just `[x]`. Decoder treats out-of-range
   indices as absent, which handles both forms.

## Out of scope (deferred)

- Path-specific URLs for "easy / medium / hard" Binairo Plus sizes —
  the supported-puzzles list links to `/binairo-plus/` and lets the
  user pick from there.
- Animated rendering of comparison-rule deductions (e.g. flashing the
  `≠` glyph that fired). Hint preview already highlights the forced
  cell; the constraint-glyph rendering is decorative.
