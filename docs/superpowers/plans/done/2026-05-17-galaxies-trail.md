# GalaxiesSolver Trail-Based Undo Plan

**Goal:** Eliminate per-recursion grid cloning in `GalaxiesSolver._search` (the two `grid.map(row => row.slice())` calls at `solver.js:883` and `:888`) by switching to trail-based undo, mirroring the technique that just landed for `NonogramSolver`.

**Architecture:** Move the grid from a method parameter to instance state (`this.grid`). Add a trail (`this.trail`) that records `(row, col, oldValue)` tuples on every assignment. `_assignPair` pushes trail entries; `_rollback(mark)` restores the grid to a saved trail length. `_search` becomes a single-instance recursion that mutates `this.grid` directly. Frontier/partial snapshots stay as `grid.map(row => row.slice())` — they're rare and cheap per snapshot, not per recursion node.

**Tech stack:** Same as before. `node:test`, `jj`.

**Scope notes:**
- The trail stores 2D `(row, col, oldValue)` tuples rather than the bit-packed encoding we used for NonogramSolver, because Galaxies cell values range from -1 to (stars.length - 1), which can exceed 2 bits when there are ≥4 stars.
- We do NOT switch to `Int8Array` storage here — separate change. Reasoning: the perf win comes from eliminating the O(rows·cols) clone per node, not from switching storage. Typed-array conversion can be a follow-up if profiling demands.
- We DO drop the `grid` parameter from internal methods (`_assignPair`, `_propagate`, `_canAssignPair`, `_candidates`, `_search`, `_regionsReachable`, `_isFilled`, `_verify`, `_matchesForbiddenPartial`, `_rememberPartial`, `_stateKey`) — they all read from `this.grid`.

### Task B.2: Add trail-based undo, eliminate per-node grid clone

Now make `_search` use trail-based undo. The two clones at the recursion loop (`solver.js:888` in current code) get replaced with mark/assign/recurse/rollback.

Steps:
1. Add `_assign(row, col, value)` and `_rollback(mark)` methods on the `GalaxiesSolver` class. Trail entry format: array `[row, col, oldValue]` triple. (2D tuple, not the bit-packed form used by NonogramSolver, because star indices can exceed 2 bits.)
2. Rewrite `_assignPair` to push two trail entries (one for the cell, one for its mirror) via `_assign`. If `_canAssignPair` returns false, `_assignPair` returns false WITHOUT pushing.
3. Rewrite `_propagate` so every write goes via `_assign`. (The only place it writes is the line `grid[r][c] = candidates[0]; ... this._assignPair(...)` — `_assignPair` already covers it; verify `_propagate` doesn't write outside of `_assignPair`.) Return type stays the same (`bool` — false on contradiction).
4. Rewrite `_search` recursion loop (currently lines ~887-893 in the moved code) from:
   ```js
   for (const starIndex of bestCandidates) {
     const next = grid.map(row => row.slice());
     const nextChanged = new Set([starIndex]);
     if (!this._assignPair(next, best.row, best.col, starIndex, nextChanged)) continue;
     const solved = this._search(next, nextChanged);
     if (solved) return solved;
   }
   ```
   to:
   ```js
   for (const starIndex of bestCandidates) {
     const mark = this.trail.length;
     const nextChanged = new Set([starIndex]);
     if (this._assignPair(best.row, best.col, starIndex, nextChanged)) {
       const solved = this._search(nextChanged);
       if (solved) return solved;
     }
     this._rollback(mark);
   }
   ```
5. Frontier-snapshot loop (lines ~882-886): same trail pattern, but capture the snapshot AFTER assigning and BEFORE rolling back:
   ```js
   for (let i = bestCandidates.length - 1; i > 0; i--) {
     const mark = this.trail.length;
     const altChanged = new Set([bestCandidates[i]]);
     if (this._assignPair(best.row, best.col, bestCandidates[i], altChanged)) {
       this._pushFrontier();
     }
     this._rollback(mark);
   }
   ```
   `_pushFrontier()` (now takes no args) reads `this.grid` and snapshots via `grid.map(row => row.slice())`.
6. When `_search` returns a solved grid (line ~892 currently `return solved` where `solved` is a grid reference): now we just `return this.grid` directly (it IS the solved state).
7. Verify trail drains on solved return: when `_search` returns the solved grid, the trail is NOT rolled back at that level — the assignment that led to the solution stays. The caller of `_search` (which is `solve()`) is responsible for output. Since `solve()` ends right after, the trail just gets garbage-collected with the solver instance. This is fine.

Verification: `npm test` → 6 pass including the trail unit test from A.2 (now passing because trail field exists and is unwound on success). Bench: 6×6 galaxies should solve in dramatically less time than baseline.

Commit: `perf(galaxies): trail-based undo in _search (no more per-node grid clones)`.
